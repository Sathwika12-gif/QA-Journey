Manual Testing Interview Questions

Here are the most commonly asked manual testing interview questions for QA testers.

1. What is the difference between verification and validation?

Verification checks whether the software is built according to specifications without executing the code. Validation checks whether the application meets user needs by executing the application.

2. What is a test scenario and how is it different from a test case?

A test scenario is a high-level description of what functionality needs to be tested. A test case is a detailed step-by-step instruction including test data and expected results.

3. Explain the bug life cycle.

The bug life cycle describes the stages a defect goes through: New → Assigned → Open/In Progress → Fixed → Retesting → Closed. If the issue persists, the tester reopens the defect.

4. What is sanity testing?

Sanity testing is performed after a bug fix or minor change to verify that the specific functionality works correctly. It is narrow and focused.

5. What will you do if a developer says “This is not a bug”?

I verify the requirement again and provide steps, screenshots, or logs to justify the issue. If behavior matches requirements, the defect can be closed; otherwise it is discussed with the BA or product owner.

6. Difference between severity and priority

Severity indicates how much the defect impacts functionality. Priority indicates how quickly the defect should be fixed.

7. What is regression testing?

Regression testing ensures that existing functionality is not affected after new changes, bug fixes, or feature additions.

8. Difference between smoke testing and sanity testing

Smoke testing verifies critical functionalities of a new build. Sanity testing verifies specific functionality after minor changes. Smoke testing is broad while sanity testing is narrow.

9. How do you write a good bug report?

  • Bug ID
  • Title
  • Description
  • Steps to reproduce
  • Expected result
  • Actual result
  • Severity and Priority
  • Environment details
  • Screenshots or logs

10. What if requirements are unclear?

Discuss with BA, product owner, or developers. Perform exploratory testing and clarify doubts with stakeholders.

11. Black box testing techniques

  • Equivalence Partitioning
  • Boundary Value Analysis
  • Decision Table Testing
  • State Transition Testing

12. Test case design techniques

  • Equivalence Partitioning
  • Boundary Value Analysis
  • Decision Table Testing
  • State Transition Testing

13. Types of non-functional testing

  • Performance Testing
  • Load Testing
  • Stress Testing
  • Security Testing
  • Usability Testing
  • Compatibility Testing

14. STLC stages

  1. Requirement Analysis
  2. Test Planning
  3. Test Case Design
  4. Test Environment Setup
  5. Test Execution
  6. Test Closure

15. Entry criteria and exit criteria

Entry criteria are conditions required before testing starts. Exit criteria are conditions required before testing stops.

16. What is exploratory testing?

Exploratory testing is an approach where testers simultaneously learn the application, design tests, and execute them.

17. Functional vs non-functional testing

Functional testing checks application features. Non-functional testing checks performance, usability, and reliability.

18. What is UAT?

User Acceptance Testing is performed by end users or business stakeholders to validate business requirements before production release.

19. Retesting vs regression testing

Retesting verifies a specific bug fix. Regression testing verifies the stability of the overall application after changes.

20. What is a test plan?

A test plan defines the testing scope, strategy, resources, schedule, and environment for the testing process.

21. Test case vs test script

A test case is a manual testing document. A test script is an automated version written in tools like Selenium or Playwright.

22. What is defect leakage?

Defect leakage occurs when a defect is missed during testing and discovered later in production.

23. What is defect density?

Defect density measures the number of defects found per module or per lines of code.

24. What is a test summary report?

A document summarizing test execution results, test coverage, and defects found during testing.

25. Test environments

  • Development
  • QA/Test
  • Pre-production
  • Production
